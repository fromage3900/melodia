# Requirements Document

## Introduction

This feature extends Melodia's existing PCG library with an advanced Escher-Nikki tier of procedural
content generation: deep impossible geometry, Infinity Nikki-inspired floating islands, organic-meets-
architectural fairy-tale forms, and the C++ custom PCG element infrastructure needed to power them.

The project already has 18 baroque PCG graphs and five Escher-tagged vanilla graphs
(`PCG_PenroseShrine`, `PCG_EscherDecks`, `PCG_DreamWalls`, `PCG_BridgeArchipelago`,
`PCG_FloatingStairways`) plus PCGEx v0.75.20.  This spec **extends** that foundation — it does not
replace it.  New graphs live in `/Game/_PROJECT/PCG/Graphs/EscherNikki/`; new C++ nodes live in
`Source/MelodiaMelusina_PROD/PCG/`.

The deliverables are:

- Six new PCG graph setups targeting Escher / Nikki archetypes
- Three reusable PCG subgraphs shared across the new archetypes
- Four C++ custom PCG element classes (one per core impossible-geometry primitive)
- An extended mesh catalog and collection assets for Escher-tier meshes
- Blueprint-exposed artist-control parameters on every new graph
- Runtime and editor-time generation parity
- Python builder extension (`escher_nikki_builder.py`) mirroring the existing baroque builder

## Glossary

- **PCG_System**: The Unreal Engine 5 Procedural Content Generation framework (plugin `PCG`).
- **PCGEx**: The PCGExtendedToolkit plugin (v0.75.20) already enabled in `Melodia.uproject`.
- **EscherNikki_Graph**: Any PCG graph asset under `/Game/_PROJECT/PCG/Graphs/EscherNikki/`.
- **PCG_Sub_Escher**: A reusable PCG subgraph under `/Game/_PROJECT/PCG/Graphs/EscherNikki/Subgraphs/`.
- **Custom_PCG_Element**: A C++ class derived from `FPCGElement` / `UPCGSettings` added to the
  `MelodiaMelusina_PROD` module under `Source/MelodiaMelusina_PROD/PCG/`.
- **GoldenRatio_Scale**: The recursive shrink factor 0.618 already used by `PCG_PenroseShrine`.
- **Module_Grid**: The canonical grid (400 cm wall width, 600 cm story height) defined in
  `PCG_SYSTEM_COHESION_2026-06-13.md` §1.
- **GravityZone**: A tagged PCG point attribute (`GravityDir : FVector`) that overrides the default
  downward axis for mesh placement and walkability checks in impossible-geometry graphs.
- **FloatingIsland**: A self-contained landmass mesh group generated by `PCG_FloatingIslandNikki`
  that combines a rock underside, layered terrain top, and architectural crown.
- **TessellationSurface**: A planar or curved surface where a repeating tile pattern tiles without
  gaps or overlaps, generated by `PCG_TessellationSurface`.
- **Staircase_Loop**: A closed stair path that returns to its starting elevation after N steps —
  the Penrose/Escher ascending staircase illusion, generated by `PCG_InfiniteStaircaseEx`.
- **Recursive_Arch**: An archway whose aperture contains a smaller identical archway, generated to
  configurable depth by `PCG_RecursiveArchEx`.
- **Builder_Script**: `Scripts/PCG/escher_nikki_builder.py`, the Python UE automation script that
  creates EscherNikki graph assets in-editor (mirrors `melodia_pcgex_builder.py`).
- **Attribute**: A named value stored per PCG point (position, rotation, scale, custom metadata).
- **Subgraph**: A `PCGSubgraphSettings` node that embeds a reusable PCG graph as a node.
- **Blueprint_Parameter**: A PCG graph parameter exposed to Blueprint / Detail panel for artist
  override at the PCG component level.
- **EscherNikki_Collection**: A `PCGCol_EscherNikki_*` mesh collection asset under
  `/Game/_PROJECT/PCG/Collections/`.

## Requirements

### Requirement 1: C++ Custom PCG Element Infrastructure

**User Story:** As a technical artist, I want C++ custom PCG nodes for impossible-geometry
primitives, so that I can build Escher-style layouts that vanilla PCG nodes cannot express.

#### Acceptance Criteria

1. THE `MelodiaMelusina_PROD` module SHALL declare a `PCG` subdirectory under
   `Source/MelodiaMelusina_PROD/PCG/` containing all Custom_PCG_Element source files.
2. THE `MelodiaMelusina_PROD.Build.cs` SHALL add `"PCG"`, `"PCGEx"`, and
   `"GeometryScripting"` to `PublicDependencyModuleNames` so that Custom_PCG_Element
   classes compile without unresolved symbol errors.
3. THE `PCGEscherStaircaseSettings` Custom_PCG_Element SHALL accept inputs: `StepCount`
   (int32, default 16), `StepHeight` (float, default 25 cm), `StepWidth` (float, default
   150 cm), `LoopRadius` (float, default 600 cm), and `Seed` (int32, default 0), and SHALL
   output a set of PCG points forming a closed Staircase_Loop whose final step position
   equals the first step position within 1 cm tolerance.
4. THE `PCGGravityZoneSettings` Custom_PCG_Element SHALL accept an input point set and a
   `GravityDir` FVector parameter, and SHALL write a `GravityDir` attribute to every output
   point so that downstream spawner nodes can orient meshes to the overridden gravity axis.
5. THE `PCGRecursiveArchSettings` Custom_PCG_Element SHALL accept inputs: `ArchWidth`
   (float, default 400 cm), `ArchHeight` (float, default 600 cm), `RecursionDepth` (int32,
   1–4), and `ScaleFactor` (float, default 0.618), and SHALL output point sets for each arch
   tier such that each tier's width equals the parent tier's width multiplied by `ScaleFactor`.
6. THE `PCGTessellationSettings` Custom_PCG_Element SHALL accept a surface input and a
   `TileShape` enum (Square, Hexagon, Penrose), and SHALL output a point set covering the
   surface with zero-gap tile placement where each tile point carries a `TileType` int32
   attribute.
7. WHEN any Custom_PCG_Element receives an out-of-range parameter (e.g. `RecursionDepth`
   outside 1–4, `StepCount` less than 3), THEN THE Custom_PCG_Element SHALL clamp the
   value to the nearest valid bound and log a warning via `UE_LOG(LogPCG, Warning, …)`.
8. WHERE the `PCG` and `PCGEx` modules are not loaded, THE Custom_PCG_Element classes SHALL
   compile without errors by guarding PCGEx-specific includes with `#if WITH_PCGEX`.

### Requirement 2: Infinite Staircase PCG Graph

**User Story:** As a level designer, I want a PCG graph that generates a visually looping
infinite staircase, so that I can place Escher-style ascending paths that players perceive
as endless without navigating an infinite space.

#### Acceptance Criteria

1. THE EscherNikki_Graph `PCG_InfiniteStaircaseEx` SHALL use `PCGEscherStaircaseSettings`
   to generate a Staircase_Loop of configurable step count.
2. THE EscherNikki_Graph `PCG_InfiniteStaircaseEx` SHALL expose Blueprint_Parameters:
   `StepCount` (int32), `StepMeshCollection` (soft object ref to EscherNikki_Collection),
   `RailingEnabled` (bool), `Seed` (int32), and `MaterialOverride` (soft object ref to
   material interface).
3. WHEN `RailingEnabled` is true, THE EscherNikki_Graph `PCG_InfiniteStaircaseEx` SHALL
   spawn railing mesh modules along both sides of every stair step using
   `PCG_Sub_Escher_Railing`.
4. WHEN `StepCount` is set to any value between 8 and 64 inclusive, THE
   `PCG_InfiniteStaircaseEx` graph SHALL generate without editor errors and produce at least
   `StepCount` spawned mesh instances.
5. THE EscherNikki_Graph `PCG_InfiniteStaircaseEx` SHALL write `ArchitecturalRole = "Stair"`
   and `Walkable = true` attributes to all stair-tread points per the Module_Grid shared
   attribute vocabulary.
6. THE EscherNikki_Graph `PCG_InfiniteStaircaseEx` SHALL support both editor-time and
   runtime generation without requiring a cook-time only flag.

### Requirement 3: Tessellating Surface PCG Graph

**User Story:** As a level designer, I want a PCG graph that tiles a repeating geometric
pattern across any surface, so that I can create Escher-inspired floor and wall tessellations
that feel hand-crafted at any scale.

#### Acceptance Criteria

1. THE EscherNikki_Graph `PCG_TessellationSurface` SHALL use `PCGTessellationSettings`
   to place tile instances across a PCG volume or surface input.
2. THE EscherNikki_Graph `PCG_TessellationSurface` SHALL expose Blueprint_Parameters:
   `TileShape` (enum: Square | Hexagon | Penrose), `TileScale` (float, default 200 cm),
   `AlternateCollection` (soft object ref), `RandomRotation` (bool), and `Seed` (int32).
3. WHEN `TileShape` is Penrose, THE EscherNikki_Graph `PCG_TessellationSurface` SHALL
   generate two tile types (fat and thin rhombi) with a `TileType` attribute value of 0 or 1
   assigned to each point so that mesh selectors can distinguish them.
4. WHEN `TileShape` is Hexagon or Square, THE EscherNikki_Graph `PCG_TessellationSurface`
   SHALL produce zero visible gaps between adjacent tiles at `TileScale` 200 cm over a 20 m × 20 m
   flat surface, verified by confirming no point-to-point spacing exceeds `TileScale × 1.05`.
5. THE EscherNikki_Graph `PCG_TessellationSurface` SHALL write `ArchitecturalRole = "Floor"`
   and `Walkable = true` to all output points by default, overridable via a `WalkabilityTag`
   Blueprint_Parameter.
6. THE EscherNikki_Graph `PCG_TessellationSurface` SHALL support both editor-time and
   runtime generation.

### Requirement 4: Recursive Arch PCG Graph

**User Story:** As a level designer, I want a PCG graph that nests identical archways inside
each other at shrinking scales, so that I can create the infinite-depth portal illusion seen
in Escher's work and fairy-tale architecture.

#### Acceptance Criteria

1. THE EscherNikki_Graph `PCG_RecursiveArchEx` SHALL use `PCGRecursiveArchSettings` with
   `RecursionDepth` 1–4 to produce layered arch point sets.
2. THE EscherNikki_Graph `PCG_RecursiveArchEx` SHALL expose Blueprint_Parameters:
   `ArchWidth` (float), `ArchHeight` (float), `RecursionDepth` (int32, 1–4),
   `ScaleFactor` (float, default 0.618), `ArchCollection` (soft object ref), and `Seed` (int32).
3. WHEN `RecursionDepth` is 3, THE EscherNikki_Graph `PCG_RecursiveArchEx` SHALL produce
   exactly 3 sets of arch points where each successive set's spawn-position bounding box width
   equals the previous set's bounding box width multiplied by `ScaleFactor` within 2% tolerance.
4. THE EscherNikki_Graph `PCG_RecursiveArchEx` SHALL write `ArchitecturalRole = "Arch"` and
   a `RecursionTier` int32 attribute (0 = outermost) to every output point.
5. THE EscherNikki_Graph `PCG_RecursiveArchEx` SHALL support both editor-time and runtime
   generation.
6. IF `ScaleFactor` is set below 0.3, THEN THE EscherNikki_Graph `PCG_RecursiveArchEx`
   SHALL clamp it to 0.3 and emit a Blueprint-visible warning via `FPCGContext::LogAndNotifyUser`.

### Requirement 5: Gravity-Defying Structure PCG Graph

**User Story:** As a level designer, I want a PCG graph that generates architecture oriented
to a configurable gravity direction, so that I can place upside-down towers, sideways bridges,
and impossible-angle walkways characteristic of Escher's gravity-defying compositions.

#### Acceptance Criteria

1. THE EscherNikki_Graph `PCG_GravityDefyingStructure` SHALL use `PCGGravityZoneSettings`
   to write `GravityDir` to all output points before spawning meshes.
2. THE EscherNikki_Graph `PCG_GravityDefyingStructure` SHALL expose Blueprint_Parameters:
   `GravityDir` (FVector, default (0,0,-1)), `StructureType` (enum: Tower | Bridge | Platform |
   Cantilever), `ModuleCollection` (soft object ref), `StoryCount` (int32, 1–8), and `Seed` (int32).
3. WHEN `GravityDir` is (0,0,1) (inverted), THE EscherNikki_Graph
   `PCG_GravityDefyingStructure` SHALL spawn all meshes with their local up-axis aligned to
   world-down within 1° angular tolerance.
4. WHEN `StructureType` is Tower, THE EscherNikki_Graph `PCG_GravityDefyingStructure` SHALL
   produce `StoryCount` stacked wall modules spaced by `StoryHeight` (600 cm) along the
   `GravityDir` axis.
5. THE EscherNikki_Graph `PCG_GravityDefyingStructure` SHALL write `GravityDir` and
   `ArchitecturalRole` attributes to all output points per the shared attribute vocabulary.
6. THE EscherNikki_Graph `PCG_GravityDefyingStructure` SHALL support both editor-time and
   runtime generation.

### Requirement 6: Floating Island PCG Graph (Infinity Nikki Aesthetic)

**User Story:** As a level designer, I want a PCG graph that generates layered floating
islands with an organic underside, whimsical terrain top, and an architectural crown, so that
I can populate dreamlike overworld zones matching Infinity Nikki's fairy-tale scale play.

#### Acceptance Criteria

1. THE EscherNikki_Graph `PCG_FloatingIslandNikki` SHALL compose three layers: a rock-cluster
   underside, a terrain-scatter top, and an optional architectural crown using
   `PCG_Sub_Escher_IslandBase`, `PCG_Sub_Escher_TerrainTop`, and the existing
   `PCG_Sub_BaroqueSpawn` subgraph respectively.
2. THE EscherNikki_Graph `PCG_FloatingIslandNikki` SHALL expose Blueprint_Parameters:
   `IslandRadius` (float, default 1200 cm), `RockCollection` (soft object ref),
   `TerrainCollection` (soft object ref), `CrownEnabled` (bool, default false),
   `CrownCollection` (soft object ref), `VerticalLayerSpread` (float, default 300 cm),
   and `Seed` (int32).
3. WHEN `CrownEnabled` is true, THE EscherNikki_Graph `PCG_FloatingIslandNikki` SHALL
   spawn crown architecture modules at the top layer of the island using `PCG_Sub_BaroqueSpawn`,
   with crown points offset vertically by `VerticalLayerSpread` above the terrain-top points.
4. WHEN `IslandRadius` is between 600 cm and 3000 cm inclusive, THE EscherNikki_Graph
   `PCG_FloatingIslandNikki` SHALL generate without editor errors and produce at least one
   instance in each of the three composition layers.
5. THE EscherNikki_Graph `PCG_FloatingIslandNikki` SHALL write a `LayerRole` string attribute
   to all output points with values `"RockBase"`, `"TerrainTop"`, or `"Crown"` so that
   post-generation filters can select individual layers.
6. THE EscherNikki_Graph `PCG_FloatingIslandNikki` SHALL support both editor-time and
   runtime generation.

### Requirement 7: Organic-Meets-Architectural Vine Tower PCG Graph

**User Story:** As a level designer, I want a PCG graph that wraps organic vine and tendril
elements around baroque tower modules, so that I can create the living-architecture fusion
central to the Infinity Nikki aesthetic.

#### Acceptance Criteria

1. THE EscherNikki_Graph `PCG_VineTowerNikki` SHALL generate a vertical tower of baroque
   wall modules and overlay organic scatter (vine, moss, petal) on the same point set using
   `PCG_Sub_Escher_OrganicOverlay`.
2. THE EscherNikki_Graph `PCG_VineTowerNikki` SHALL expose Blueprint_Parameters:
   `TowerHeight` (float, default 2400 cm), `TowerRadius` (float, default 400 cm),
   `VineDensity` (float 0.0–1.0, default 0.5), `OrganicCollection` (soft object ref),
   `BaroqueCollection` (soft object ref), and `Seed` (int32).
3. WHEN `VineDensity` is 0.0, THE EscherNikki_Graph `PCG_VineTowerNikki` SHALL produce
   zero vine/organic instances while still generating full tower structure modules.
4. WHEN `VineDensity` is 1.0, THE EscherNikki_Graph `PCG_VineTowerNikki` SHALL produce
   at least one organic instance per baroque wall module point.
5. THE EscherNikki_Graph `PCG_VineTowerNikki` SHALL write `ArchitecturalRole = "Tower"`
   to baroque-tier points and `ArchitecturalRole = "Organic"` to vine-tier points.
6. THE EscherNikki_Graph `PCG_VineTowerNikki` SHALL support both editor-time and
   runtime generation.

### Requirement 8: Reusable Escher Subgraphs

**User Story:** As a technical artist, I want three composable PCG subgraphs shared across
the Escher-Nikki graphs, so that I can recombine primitives into new layouts without
duplicating node chains.

#### Acceptance Criteria

1. THE PCG_Sub_Escher `PCG_Sub_Escher_Railing` SHALL accept an input path of points and
   SHALL spawn railing mesh modules spaced at `ModuleWidth` (400 cm) intervals along the
   path using a `RailingCollection` parameter, outputting all spawned instances.
2. THE PCG_Sub_Escher `PCG_Sub_Escher_IslandBase` SHALL accept a center point and
   `IslandRadius` float, and SHALL output a point cloud distributed beneath the center
   point in a roughly hemispherical underside shape using a noise-displaced sphere sample.
3. THE PCG_Sub_Escher `PCG_Sub_Escher_OrganicOverlay` SHALL accept an input point set and
   a `Density` float (0.0–1.0), SHALL apply a PCGEx noise-based filter to cull points
   according to `Density`, and SHALL spawn organic mesh instances on surviving points.
4. THE PCG_Sub_Escher `PCG_Sub_Escher_TerrainTop` SHALL accept a center point and
   `IslandRadius` float, and SHALL output a point cloud on a gently curved dome surface
   above the center using PCGEx surface sampling.
5. ALL PCG_Sub_Escher subgraphs SHALL expose a `Seed` Blueprint_Parameter for deterministic
   variation, matching the convention of all `PCG_Sub_Baroque*` subgraphs.
6. ALL PCG_Sub_Escher subgraphs SHALL be saved under
   `/Game/_PROJECT/PCG/Graphs/EscherNikki/Subgraphs/` following the naming convention
   `PCG_Sub_Escher_<Name>` with no `_BS` suffix.

### Requirement 9: EscherNikki Mesh Collections

**User Story:** As a technical artist, I want dedicated PCGEx mesh collections for
Escher-tier geometry, so that every EscherNikki_Graph references collections rather than
inline mesh lists, consistent with the project's baroque collection convention.

#### Acceptance Criteria

1. THE `baroque_mesh_catalog.json` file SHALL be extended with an `escher_nikki` top-level
   key containing at least four named collection entries: `PCGCol_EscherNikki_Stairs`,
   `PCGCol_EscherNikki_Tiles`, `PCGCol_EscherNikki_Organic`, and
   `PCGCol_EscherNikki_IslandRocks`.
2. EACH collection entry in `escher_nikki` SHALL list mesh paths under
   `/Game/_PROJECT/MelusinasHouse/` or `/Game/_PROJECT/houseassets/` where matching
   meshes exist (e.g. `SM_SpiralStair001` for Stairs, `SM_ceilingsquare` for Tiles),

   any mesh type not yet authored.
3. THE `escher_nikki_builder.py` script SHALL read collection entries from `baroque_mesh_catalog.json`
   under the `escher_nikki` key and SHALL not duplicate entries already in the `collections` key.
4. WHEN an EscherNikki_Graph references a `PCGCol_EscherNikki_*` collection and the
   collection asset does not exist in the editor, THEN THE EscherNikki_Graph SHALL log a
   missing-collection warning rather than silently producing empty output.

### Requirement 10: Python Builder Script Extension

**User Story:** As a technical artist, I want a `escher_nikki_builder.py` script that mirrors
the existing `melodia_pcgex_builder.py` pattern, so that I can create or recreate all
EscherNikki graph assets via a single Python call from the Unreal Editor output log.

#### Acceptance Criteria

1. THE Builder_Script `escher_nikki_builder.py` SHALL expose a `build_all()` function that
   creates all six EscherNikki_Graphs and all four PCG_Sub_Escher subgraphs in
   `/Game/_PROJECT/PCG/Graphs/EscherNikki/`.
2. THE Builder_Script `escher_nikki_builder.py` SHALL expose a `build_graph(name: str)`
   function that creates a single named graph, raising `KeyError` for unknown names.
3. THE Builder_Script `escher_nikki_builder.py` SHALL reuse helper functions from
   `melodia_pcgex_builder.py` (imported by file path, same pattern as the existing script)
   rather than duplicating `create_graph_asset`, `add_node`, `wire_chain`, and
   `configure_spawner_from_collection`.
4. WHEN `build_all()` is called and PCGEx settings classes are not loaded, THE Builder_Script
   SHALL log a warning and continue building graphs using vanilla PCG fallback nodes,
   identical to the behaviour in `melodia_pcgex_builder.py`.
5. THE Builder_Script `escher_nikki_builder.py` SHALL be saved at
   `Scripts/PCG/escher_nikki_builder.py` and SHALL include a usage docstring in the same
   format as `melodia_pcgex_builder.py`.
6. FOR ALL graph names returned by `build_all()`, parsing the returned list and calling
   `build_graph(name)` individually SHALL return the same asset path as `build_all()`
   (round-trip consistency between bulk and single-graph build modes).

### Requirement 11: Blueprint-Exposed Parameters and Artist Control

**User Story:** As a level designer, I want every EscherNikki_Graph to expose clearly named
Blueprint parameters in the PCG component detail panel, so that I can tune generation without
opening the graph editor.

#### Acceptance Criteria

1. EVERY EscherNikki_Graph SHALL expose a minimum set of Blueprint_Parameters including at
   least `Seed` (int32) and one mesh collection reference, consistent with the `*Ex` baroque
   graph convention in PCG_SYSTEMS_GUIDE.md §4.3.
2. EVERY EscherNikki_Graph SHALL expose `Seed` with a default value of 0 and SHALL produce
   deterministic, identical output when called with the same `Seed` and other parameters.
3. WHEN a Blueprint_Parameter is changed on a placed PCG component at editor-time, THE
   PCG_System SHALL regenerate the graph within the same editor session without requiring a
   level reload.
4. WHEN a Blueprint_Parameter `Seed` is incremented by 1, THE EscherNikki_Graph SHALL
   produce a visually distinct variation (different point positions or rotations) rather than
   identical output to `Seed 0`, for all graphs containing a stochastic element.
5. THE Blueprint_Parameters of all EscherNikki_Graphs SHALL use descriptive display names
   (e.g. "Step Count" not "StepCount") following Unreal Engine meta specifier
   `DisplayName` conventions.

### Requirement 12: Runtime and Editor-Time Generation Parity

**User Story:** As a programmer, I want all EscherNikki_Graphs to generate correctly at both
editor-time and runtime, so that zones can be procedurally streamed during gameplay without
separate cook-time baking.

#### Acceptance Criteria

1. EVERY EscherNikki_Graph SHALL have the PCG graph property `GenerationTrigger` set to
   `GenerateOnLoad` so that the graph runs both in the editor viewport and at PIE/runtime
   without manual regeneration.
2. WHEN an EscherNikki_Graph is placed in a level that is opened in Play-In-Editor mode,
   THE PCG_System SHALL generate the graph's output within 3 seconds of level load on a
   reference machine running UE5.4 or later.
3. WHEN a Custom_PCG_Element's `Execute` function is called on the game thread, THE
   Custom_PCG_Element SHALL complete execution within 16 ms for input sets of up to 512 points,
   measured via `SCOPE_CYCLE_COUNTER(STAT_PCGEscherElement)`.
4. THE Custom_PCG_Element classes SHALL be thread-safe: their `Execute` functions SHALL NOT
   write to shared mutable state outside the output `FPCGTaggedData` they return.
5. WHERE editor-only includes are required by a Custom_PCG_Element (e.g. for detail panel
   customization), THE Custom_PCG_Element SHALL guard those includes with
   `#if WITH_EDITOR … #endif` so that shipping builds compile without editor dependencies.

### Requirement 14: Shared Attribute Vocabulary and Cohesion

**User Story:** As a level designer, I want all new EscherNikki_Graphs to write the same
shared point attributes as the existing baroque graphs, so that downstream Blueprint filters,
navigation systems, and material overrides work without graph-specific branching.

#### Acceptance Criteria

1. ALL EscherNikki_Graphs and PCG_Sub_Escher subgraphs SHALL write `ArchitecturalRole`
   (string), `Walkable` (bool), and `Seed` (int32) attributes to output points where
   semantically applicable, matching the shared vocabulary in
   PCG_SYSTEM_COHESION_2026-06-13.md §3.
2. EVERY EscherNikki_Graph that produces navigable output SHALL run the existing
   `PCG_Sub_WalkabilityCheck` subgraph on navigable-tagged points, enforcing ≤ 50 cm max
   step height and ≥ 200 cm minimum path width per the cohesion document §4.
3. THE `ModuleWidth` and `StoryHeight` parameters on EscherNikki_Graphs SHALL default to
   400 cm and 600 cm respectively, matching Module_Grid in the cohesion document.
4. WHEN an EscherNikki_Graph is used in the same level as an existing baroque `*Ex` graph
   and both are merged via a PCGEx Merge Points node, THE PCG_System SHALL produce a
   combined output with consistent `ArchitecturalRole` and `Walkable` attributes on all
   points without type conflicts.
